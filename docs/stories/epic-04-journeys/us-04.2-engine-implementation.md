# User Story 04.2: Motor de Execução de Jornadas

## 🎯 Objetivo (O que e por quê)
Dar "vida" ao editor visual. O motor deve ser capaz de detectar gatilhos (ex: novo lead), inscrever o lead na jornada e processar cada passo (enviar emails, esperar tempo, validar condições) de forma assíncrona e resiliente.

## 🧠 Contexto de Expert
Uma jornada de sucesso não é apenas um desenho; é um sistema determinístico de estados. Precisamos garantir que:
1.  **Resiliência**: Se o servidor cair, o lead não pode "sumir" da jornada.
2.  **Escalabilidade**: Processar milhares de leads em paralelo sem gargalos.
3.  **Observabilidade**: Cada passo deve ser logado para auditoria e métricas.

## 📝 Fluxo de Experiência/UX (Passo a passo do usuário)
1.  O Usuário ativa a jornada no Editor.
2.  Um novo lead chega via importação ou API.
3.  O sistema identifica que existe uma jornada ativa com gatilho "Novo Lead".
4.  O lead é inscrito (`journey_enrollments`).
5.  O executor processa o primeiro nó após o trigger.
6.  Se for um nó de "Espera", o sistema agenda a próxima execução.
7.  O usuário vê as métricas de "leads no passo X" subindo em tempo real.

## 📝 Requisitos Técnicos
- **Normalização**: Ao ativar uma jornada, converter `canvas_data` em registros nas tabelas `journey_steps` e `journey_edges` para facilitar a execução sequencial.
- **Trigger Handler**: Criar uma função intermediária `enrollInRelevantJourneys(event, leadData)` que seja chamada nos pontos de entrada de dados (API de Leads).
- **Step Executor**: Um serviço (que pode ser disparado via Edge Function ou Worker) que recebe um `enrollment_id` e processa o próximo passo disponível.
- **Wait Logic**: Utilizar `pg_cron` (se disponível no Supabase) ou uma fila de agendamento externa para processar passos após X tempo.
- **RLS**: Garantir que um tenant nunca execute uma jornada de outro.

## ✅ Critérios de Aceite
- [ ] No `handleSave('active')`, o sistema normaliza os dados do canvas para as tabelas relacionais.
- [ ] Implementar trigger para `lead_created` integrado à API de importação/criação.
- [ ] Novo lead entrando no sistema é automaticamente inscrito em jornadas ativas.
- [ ] O sistema processa o primeiro passo imediatamente (Ex: Enviar Email de Boas-vindas).
- [ ] O nó de "Wait" agenda corretamente a continuação do fluxo.
- [ ] O nó de "Condition" ramifica o fluxo corretamente baseado no status.
- [ ] Logs detalhados em `journey_step_logs` para cada lead.
- [ ] Atualização automática do campo `total_entered` e `total_completed` nos steps.
