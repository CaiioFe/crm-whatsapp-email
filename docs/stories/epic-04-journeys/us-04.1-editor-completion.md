# US 04.1 — Finalização do Editor de Jornadas (Save & Load)

## 🎯 Objetivo (O que e por quê)
Finalizar a persistência de dados do Editor de Jornadas, permitindo que o arquiteto de fluxos salve suas configurações no banco de dados, recupere jornadas salvas e ative a automação.

## 🧠 Contexto de Expert
Uma jornada sem persistência é apenas um desenho. Para o marketing, a agilidade de salvar rascunhos e testar variações é crucial. Tecnicamente, precisamos garantir que o `canvas_data` seja salvo integralmente para reconstruir o React Flow no carregamento.

## 📝 fluxo de Experiência/UX
1. O usuário desenha o fluxo no canvas.
2. Clica em "Salvar" e o sistema persiste no Supabase via API.
3. O status permanece como `draft`.
4. Ao clicar em "Ativar", o status muda para `active` (exige permissão manager/admin).
5. O sistema valida se há ao menos um trigger e um caminho de saída.

## 📝 Requisitos Técnicos
- Endpoint API `/api/journeys/save` (implementado).
- Hook/Fetch no frontend para enviar `nodes` e `edges`.
- Tratamento de estados (Saving..., Saved!).
- Implementação de Feature Flags (`journey_editor_enabled`).
- RBAC validation no botão de Ativar.

## ✅ Critérios de Aceite
- [x] Implementar chamada de API `POST` ao clicar no botão "Salvar".
- [x] Carregar jornada se houver `id` na URL (modo edição).
- [x] Botão "Ativar" atualiza status para `active` e valida integridade.
- [x] Feedback visual de rascunho salvo (Toast).
- [x] Garantir que o `tenant_id` seja respeitado em todas as operações.
- [x] Exibir métricas reais de entrada/saída se o fluxo já estiver ativo.
